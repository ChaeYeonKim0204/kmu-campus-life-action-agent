# KMU 캠퍼스 생활 액션 에이전트 서비스 기획서

> 문서 목적: 현재 구현된 서비스의 제품 방향, 핵심 기능, 사용자 흐름, 운영 정책을 한눈에 공유하기 위한 기획 문서  
> 작성일: 2026-05-22  
> 대상: 기획, 프론트엔드, 백엔드, 에이전트/RAG, QA 담당자  
> 현재 개발 주소: `http://127.0.0.1:5173/`

## 1. 서비스 개요

### 1.1 서비스명

**KMU 캠퍼스 생활 액션 에이전트**

### 1.2 한 줄 설명

국민대학교 학생이 학사·행정·캠퍼스 생활 질문을 입력하면, 공식 자료를 근거로 답변하고 다음 행동까지 이어주는 한국어 기반 캠퍼스 생활 보조 에이전트입니다.

### 1.3 서비스가 해결하는 문제

학생은 학사 행정 절차를 찾을 때 여러 공식 페이지, 공지, 학사일정, 부서 안내를 직접 오가야 합니다. 특히 출석인정, 휴학/복학, 수강신청, 증명서, 학생증, 등록금, 장학금처럼 절차와 제출처가 함께 필요한 질문은 단순 검색만으로 해결하기 어렵습니다.

이 서비스는 다음 문제를 줄이는 것을 목표로 합니다.

- 공식 자료가 흩어져 있어 필요한 절차를 빠르게 찾기 어려움
- “무엇을 해야 하는지”는 알아도 “어디에, 언제까지, 어떤 서류로” 진행해야 하는지 불명확함
- 개인정보를 넣어야 할 것 같은 불안감이 있음
- 챗봇 답변의 근거를 확인하기 어려움
- 답변 이후 신청서, 체크리스트, 문의문 작성으로 바로 이어지지 않음

### 1.4 핵심 가치

- **근거 기반 답변:** 공식 자료 chunk와 citation을 기반으로 답변합니다.
- **다음 행동 연결:** 안내에서 끝나지 않고 신청서 초안, 체크리스트, 문의문 작성으로 이어집니다.
- **개인정보 보호:** 학번, 주민번호, 연락처, 성적, 포털 ID/PW 등 민감정보 입력을 차단합니다.
- **학생 친화 UI:** 메타버스 캠퍼스 화면에서 구역을 탐색하고, NPC/퀘스트 방식으로 행정 절차를 이해합니다.
- **검증 가능한 처리 과정:** 분류, 검색, guard, tool 실행 로그와 근거 출처를 화면에서 확인할 수 있습니다.

## 3. 현재 구현 범위

### 3.1 사용자 화면

현재 프론트엔드는 Vite + React 기반이며, 개발 서버 기준 `http://127.0.0.1:5173/`에서 확인할 수 있습니다.

주요 화면 구성은 다음과 같습니다.

- 메타버스 캠퍼스 메인 화면
- 캠퍼스 건물별 NPC/퀘스트 탐색
- 질문 입력 및 답변 표시
- 추천 질문 버튼
- 신청서/체크리스트 작성 액션 폼
- 출처 패널
- 처리 상태 패널
- 툴 로그 패널
- 관리자 대시보드

### 3.2 메타버스 캠퍼스 UI

캠퍼스 지도를 중심으로 건물별 행정/학사 주제를 배치했습니다.

| 건물 | 주요 역할 |
| --- | --- |
| 본부관 | 출석인정, 학사 행정, 신청서 작성 |
| 복지관 | 휴학/복학, 종합민원, 학생지원 |
| 성곡도서관 | 학생증, 도서관 이용, 좌석/연체 관련 |
| 이캠퍼스 센터 | eCampus, 온라인 강의, 계정/동기화 문제 |
| 북악관 | 수강신청, 장학금, 증명서, 학사 일정 |
| 공학관 | 학과 행정, 실습실, 졸업요건, 폐강 관련 |

최근 UI 점검을 통해 메타버스 화면은 웹페이지 기준으로 확대했고, 다음 겹침 문제를 개선했습니다.

- 지도와 채팅 패널의 좁은 화면 겹침 완화
- 미니맵과 채팅 패널 분리
- 건물 툴팁과 왼쪽 메뉴 겹침 해소
- 상단 HUD와 타이틀 배치 정리
- 오른쪽 건물 배치가 화면 끝에 밀착되지 않도록 조정
- 중복 플로팅 버튼 숨김 처리

### 3.3 답변/액션 기능

현재 서비스는 다음 기능을 제공합니다.

- 질문 유형 분류
- 개인정보 guard
- 공식 자료 hybrid 검색
- citation 기반 답변 조립
- 체크리스트 생성
- 문의처 추천
- 일정/마감일 계산
- 다음 행동 추천
- 출석인정신청서 등 문서 초안 작성
- 휴학/복학 체크리스트 작성
- 졸업요건 간이 진단
- 수강계획 추천

### 3.4 졸업 도우미 프로토타입

`test/` 폴더에는 메인 서비스와 별도로 실행되는 Streamlit 기반 졸업 도우미 프로토타입이 있습니다. 이 기능군은 졸업예정자의 요구를 더 깊게 다루기 위한 실험 축입니다.

| 기능 | 내용 |
| --- | --- |
| 성적증명서 입력 | PDF 업로드 또는 수동 입력으로 학과, 입학연도, 이수학점, GPA, 과목 목록을 구성 |
| 성적증명서 파싱 | 텍스트 PDF는 `pdfplumber`, 이미지 기반 PDF는 GPT-4o Vision OCR 사용 |
| 졸업요건 구조화 | 2025 요람 별표5/별표6에서 학과별 졸업 최저이수학점, 공학인증 요건, 이수구분 코드 추출 |
| 요람 RAG 인덱싱 | 2025 국민대학교 요람 PDF를 ChromaDB에 인덱싱하고 페이지/섹션 메타데이터 유지 |
| 졸업 진단 | 구조화 JSON과 요람 RAG를 함께 사용해 졸업 가능 여부와 부족 학점을 분석 |
| 대체 이수 탐색 | 폐강 또는 수강 실패 과목의 대체 가능 과목 후보 탐색 |
| 마이크로디그리 발굴 | 이수 과목 기반으로 취득 가능성이 높은 마이크로디그리/소학위 트랙 추천 |
| 졸업 후 체크리스트 | 졸업 전, 졸업 직전, 졸업 후 즉시, 졸업 후 장기 단계별 할 일 생성 |
| 직무 역량 번역기 | 이수 과목을 희망 직무의 역량 키워드와 자기소개서 문장 예시로 변환 |
| 결과 다운로드 | 교학팀 확인 요청서와 직무 역량 분석 결과를 TXT로 다운로드 |

단, 이 프로토타입은 성적증명서와 개인 성적 정보를 직접 다루므로 메인 서비스의 개인정보 guard 정책과 그대로 결합하면 안 됩니다. 메인 서비스 통합 시에는 성적표 원본 업로드 대신 학생이 직접 요약한 비식별 학점 정보만 받는 형태로 재설계해야 합니다.

## 4. 핵심 사용자 여정

### 4.1 여정 A: 질문 → 근거 기반 답변

```text
서비스 접속
→ 메타버스 캠퍼스 화면 확인
→ 질문 입력 또는 건물/추천 질문 선택
→ 개인정보 guard 실행
→ 질문 유형 분류
→ 공식 자료 검색
→ 체크리스트/문의처/기한 tool 실행
→ citation 포함 답변 표시
→ 근거 출처 확인
→ 필요한 경우 다음 행동 선택
```

예시 질문:

- `예비군 때문에 결석하는데 뭐 해야 해?`
- `5월 15일에 예비군 때문에 결석하는데 출석인정 언제까지 내야 해?`
- `질병휴학 하려면 뭐 필요해?`
- `수강신청 완료됐는지 어디서 확인해?`
- `졸업예정증명서 어디서 뽑아?`

### 4.2 여정 B: 답변 → 신청서/체크리스트 초안

```text
답변 화면에서 추천 액션 선택
→ /actions/start 호출
→ 필요한 slot 질문 표시
→ 사용자가 항목 입력
→ 입력값 개인정보 guard
→ /actions/continue 호출
→ 공식 근거 기반 문서 초안 생성
→ 제출 전 체크리스트와 유의사항 확인
```

출석인정신청서 예시 slot:

- 결석일
- 결석 사유
- 수업명
- 교강사명
- 증빙서류
- 제출 예정일

단, 학번, 주민번호, 연락처, 성적표 원본, 포털 ID/PW는 입력받지 않습니다.

### 4.3 여정 C: 캠퍼스 탐색 → 퀘스트 수행

```text
지도에서 건물 선택
→ 해당 건물 NPC/퀘스트 확인
→ 관련 추천 질문 선택
→ 답변 확인
→ 필요한 액션 수행
→ 퀘스트 완료 상태 반영
```

이 여정은 단순 챗봇보다 진입 장벽이 낮고, 학생이 학교 업무를 “장소와 담당 역할” 중심으로 이해하도록 돕습니다.

## 5. 주요 기능 명세

### 5.1 질문 답변

| 항목 | 내용 |
| --- | --- |
| 입력 | 자연어 한국어 질문 |
| 처리 | guard → classifier → retriever → planner/tools → answer builder |
| 출력 | 답변, 근거 출처, tool 로그, 추천 액션 |
| 핵심 조건 | 공식 근거가 없으면 절차성 답변을 만들지 않음 |

### 5.2 개인정보 guard

| 차단 대상 | 예시 |
| --- | --- |
| 학번 | 학생 식별 번호 |
| 주민번호 | 주민등록번호 |
| 연락처 | 휴대폰 번호, 전화번호 |
| 성적 | 성적표 원본, GPA, 개인 성적 |
| 계정 정보 | 포털 ID/PW, 비밀번호 |

guard는 `/ask`와 `/actions/continue` 양쪽에서 실행됩니다.

### 5.3 공식 근거 검색

검색 데이터의 source of truth는 `data/processed/chunks.jsonl`입니다. Chroma vector DB는 가속 경로이며, 사용 불가해도 JSONL keyword 검색 fallback으로 서비스가 동작해야 합니다.

공식 source tier는 다음과 같습니다.

| Tier | 출처 | 역할 |
| --- | --- | --- |
| 1 | 규정관리시스템 | 제도적 가능 여부 |
| 2 | 학사안내 | 절차, 신청경로, 필요서류 |
| 3 | 학생지원/대학생활 안내 | 증명서, 학생증, 병무, 상담 |
| 4 | 학사일정 | 신청기간, 마감일 |
| 5 | 공지사항 | 학기별 변경사항 |
| 6 | 요람/규정집 | 졸업요건, 교육과정 |
| 7 | 대학조직 | 문의처 라우팅 |
| 8 | SWELL 공개 게시판 | 공개 신청 안내 |

### 5.4 액션 플로우

지원 액션은 다음 방향으로 구성됩니다.

- 출석인정신청서 초안
- 휴학/복학 체크리스트
- 수강신청 확인 안내
- 증명서 발급 안내
- 졸업요건 간이 진단
- 수강계획 추천
- 문의문 초안

각 액션은 slot schema를 기반으로 필요한 정보만 순차적으로 입력받습니다.

### 5.5 관리자/운영 기능

관리자 대시보드는 서비스 운영과 데이터 상태 점검을 위한 보조 화면입니다.

관리 대상:

- API 상태
- 공식자료 수집 실행
- live refresh 실행
- source 목록 확인
- vector/keyword retriever 상태
- LLM 보조 기능 상태

운영 기능은 향후 인증/권한 분리가 필요합니다.

## 6. 시스템 구조

### 6.1 전체 흐름

```text
사용자 질문
→ FastAPI /ask
→ agent.guard.inspect_privacy
→ agent.classifier.classify_issue
→ retriever.HybridRetriever.search
→ agent.guard.require_sources
→ agent.planner.suggest_actions
→ agent.answer_builder.build_final_answer
→ Frontend 렌더링
```

### 6.2 액션 흐름

```text
사용자 액션 선택
→ POST /actions/start
→ agent.action_state.start_action
→ slot 질문 반환
→ POST /actions/continue
→ slot 값 개인정보 guard
→ tools.document_drafter.draft_action_document
→ 공식 근거 기반 초안 반환
```

### 6.3 데이터 수집 흐름

```text
공식 공개 소스
→ crawler
→ parser
→ chunker
→ data/processed/chunks.jsonl
→ Chroma vector index optional
→ HybridRetriever.reload
```

학교 서버 보호를 위해 crawler는 보수적으로 동작합니다.

- 요청 간 랜덤 지연
- 실행당 최대 페이지 수 제한
- ingest cooldown
- 동시 ingest lock
- ETag/Last-Modified 기반 조건부 요청
- 네트워크 실패 시 curated fallback text 사용
- 응답에 network/fallback 상태 노출

### 6.4 졸업 도우미 프로토타입 흐름

```text
성적증명서 PDF 또는 수동 입력
→ test/2_agents.py TranscriptInfo 구성
→ test/graduation_requirements.json 구조화 기준 조회
→ test/chroma_db 요람 RAG 검색
→ GPT-4o 기반 졸업/대체이수/마이크로디그리/직무역량 분석
→ Streamlit UI 렌더링 및 TXT 다운로드
```

이 흐름은 현재 FastAPI `/ask` 파이프라인 밖에 있으며, 메인 서비스 통합 대상이라기보다 졸업 특화 기능 검증용 프로토타입입니다.

## 8. UX 원칙

### 8.1 학생 화면 원칙

- 기술 용어보다 “지금 무엇을 하면 되는지”를 먼저 보여줍니다.
- 공식 근거는 숨기지 않고, 필요할 때 펼쳐 확인할 수 있게 합니다.
- 개인정보 입력을 유도하지 않습니다.
- 질문 입력 이후 불필요한 모달을 자동으로 띄우지 않습니다.
- 메타버스 화면은 장식이 아니라 탐색과 액션 진입의 역할을 가져야 합니다.

### 8.2 메타버스 화면 원칙

- 웹페이지 뷰포트 기준으로 화면을 최대한 넓게 사용합니다.
- 지도, 채팅, 미니맵, 메뉴, 툴팁이 서로 주요 조작 영역을 가리지 않아야 합니다.
- 좁은 화면에서는 타이틀과 HUD 정보를 축약해 조작성을 우선합니다.
- 건물별 NPC/퀘스트는 실제 행정 주제와 연결되어야 합니다.

### 8.3 답변 원칙

- 절차성 문장에는 citation을 붙입니다.
- 공식 근거가 없는 경우 추측으로 안내하지 않습니다.
- 답변은 요약, 해야 할 일, 필요서류, 제출처, 기한, 문의처 중심으로 구조화합니다.
- 최종 제출 전 공식 포털/담당 부서 확인이 필요하다는 점을 안내합니다.

## 9. 정책 및 가드레일

반드시 유지해야 하는 정책은 다음과 같습니다.

- 학번, 주민번호, 연락처, 성적표 원본, 포털 ID/PW는 수집하지 않습니다.
- 공식 chunk가 없는 절차성 답변은 생성하지 않습니다.
- 로그인 이후 ON국민, SWELL 개인 화면에는 접근하지 않습니다.
- 에브리타임 등 비공식 서비스는 자동 크롤링하지 않습니다.
- LLM을 연결하더라도 최종 답변은 검색된 공식 chunk와 citation 계약을 유지해야 합니다.
- 서비스는 실제 행정 처리를 대신하지 않으며, 제출은 사용자가 공식 채널에서 직접 수행합니다.

## 14. MVP 완료 기준

현재 서비스의 MVP 완료 기준은 다음과 같습니다.

- 학생이 한국어 질문을 입력하면 공식 근거 기반 답변을 받을 수 있음
- 답변에 citation과 근거 출처가 표시됨
- 개인정보 입력 시 차단됨
- 출석인정, 휴학/복학, 수강신청, 증명서, 문의처 관련 핵심 질문을 처리할 수 있음
- 추천 액션을 통해 문서 초안 또는 체크리스트 작성으로 이어질 수 있음
- 메타버스 캠퍼스 화면이 데스크톱과 좁은 웹 뷰포트에서 깨지지 않음
- 테스트 스위트와 프론트엔드 빌드가 통과함

## 15. 결론

KMU 캠퍼스 생활 액션 에이전트는 단순 질의응답 챗봇이 아니라, 공식 근거 검색과 실제 행정 행동을 연결하는 학생용 실행 보조 서비스입니다. 현재 구현은 RAG 파이프라인, 개인정보 guard, citation, 액션 플로우, 메타버스형 UI까지 MVP의 핵심 골격을 갖춘 상태입니다.

다음 단계에서는 학생 화면의 정보 구조를 더 명확히 하고, 관리자 기능의 접근 제어와 데이터 품질 운영 체계를 강화하는 것이 중요합니다.
